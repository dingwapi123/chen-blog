# 个人技术博客 + 独立 CMS 技术方案 V1.0.1

## 1. 项目概述

本项目是一个单管理员维护的个人技术博客，包含一个面向访客的博客前台和一个独立的内容管理后台（CMS）。

V1.0.1 保持安全、稳定的内容发布链路：管理员可以以邮箱密码登录 CMS，管理文章、分类、标签和正文图片；访客可以浏览文章列表、文章详情、分类与标签页面。博客前台提供 SSR、页面元信息、RSS 和 Sitemap。

项目采用渐进式建设。未进入当前版本的功能会在相应技术章节说明，不单独建设一套“待办功能清单”。

## 2. 总体架构

```text
                         ┌──────────────────────────────┐
访客 ───────────────────→│ blog-web（Nuxt 4）            │
                         │ SSR、SEO、RSS、Sitemap、ISR    │
                         └──────────────┬───────────────┘
                                        │ publishable key
                                        ▼
                         ┌──────────────────────────────┐
                         │ Supabase                      │
                         │ Auth / PostgreSQL / Storage   │
                         │ RLS 是最终权限边界             │
                         └──────────────▲───────────────┘
                                        │ 管理员 JWT
                         ┌──────────────┴───────────────┐
管理员 ─────────────────→│ cms-admin（Vue 3 SPA）         │
                         │ 草稿、分类、标签、图片 CRUD    │
                         └──────────────┬───────────────┘
                                        │ 仅发布操作
                         ┌──────────────▼───────────────┐
                         │ blog-web 的 Nitro API          │
                         │ POST /api/admin/posts/:id/publish │
                         └──────────────────────────────┘
```

### 架构原则

1. **Supabase 是唯一在线数据源**：文章 Markdown、分类、标签和媒体记录都在 Supabase；不同时维护 `content/*.md` 作为第二份主数据。
2. **RLS 是安全边界**：CMS 可以直接调用 Data API，但数据库必须拒绝不符合权限的请求。
3. **复杂流程才经过 Nitro**：V1.0.0 只将“发布文章”集中到 Nitro；普通草稿 CRUD 直接走 Supabase，避免过早搭建大量 BFF 接口。
4. **浏览器永不持有服务端密钥**：浏览器仅使用 Supabase publishable key；`service_role`、邮件或 AI 密钥只允许可信服务端使用。V1.0.0 中 `service_role` 仅由 Nitro 发布接口在完成 token 与 owner 校验后使用。
5. **缓存可预测优先于即时生效**：V1.0.0 只接受 10 分钟缓存延迟，不增加 webhook 刷新或第二层 CDN 缓存。

## 5. 身份与管理员初始化

V1.0.0 只有一个角色：`owner`。

### 必须遵守的初始化流程

1. 在 Supabase Auth 设置中关闭公开注册。
2. 在 Supabase Dashboard 手动创建唯一的邮箱密码用户。
3. 使用受控 SQL 为该用户的 UID 插入一条 `profiles` 记录，角色为 `owner`。
4. CMS 只提供登录、登出和修改当前密码；不出现注册、邀请、角色管理或用户管理页面。

`profiles.role` **不能有 `owner` 默认值**，也不能由浏览器插入或修改。仅在管理端创建账号时写入一次。

```sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  display_name varchar(100),
  role varchar(20) not null check (role in ('owner')),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- 在 Dashboard 创建 Auth 用户后，以数据库管理权限执行：
insert into public.profiles (id, display_name, role)
values ('<AUTH_USER_UUID>', 'Chen', 'owner');
```

## 6. 数据模型

V1.0.0 只建立以下核心表。

| 表 | 作用 |
| --- | --- |
| `profiles` | 唯一管理员的身份与角色。 |
| `posts` | Markdown 文章、发布状态和软删除状态。 |
| `categories` | 文章分类。 |
| `tags` | 文章标签。 |
| `post_tags` | 文章与标签的多对多关系。 |
| `media` | Storage 文件的元数据和引用目标。 |

迁移的建表顺序为：`profiles` → `categories`、`tags` → `media` → `posts` → `post_tags`，以满足外键依赖。

### `posts` 的关键字段与约束

```sql
create table public.posts (
  id uuid primary key default gen_random_uuid(),
  title varchar(200) not null,
  slug varchar(200) not null unique,
  summary text not null default '',
  content text not null default '',
  cover_media_id uuid references public.media(id) on delete set null,
  category_id uuid references public.categories(id) on delete set null,
  status varchar(20) not null default 'draft'
    check (status in ('draft', 'published', 'archived')),
  published_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  deleted_at timestamptz,
  constraint published_posts_have_timestamp
    check (status <> 'published' or published_at is not null)
);

create index posts_public_listing_idx
  on public.posts (published_at desc)
  where status = 'published' and deleted_at is null;
```

不创建 `scheduled_at`。发布时间由数据库触发器强制维护：新建的已发布文章或状态首次切换为 `published` 时，触发器将 `published_at` 写为 `now()`；已发布文章不得单独修改该字段；文章重新归档或转回草稿时将其清空，下一次发布再次写入当前时间。因此客户端无法将文章设置为未来发布时间。

```sql
create schema if not exists private;

create function private.enforce_post_publication_fields()
returns trigger
language plpgsql
set search_path = ''
as $$
begin
  if new.status = 'published' then
    if tg_op = 'INSERT' then
      new.published_at := now();
    elsif old.status is distinct from 'published' then
      new.published_at := now();
    elsif new.published_at is distinct from old.published_at then
      raise exception 'published_at cannot be edited directly';
    end if;
  else
    new.published_at := null;
  end if;

  return new;
end;
$$;

revoke all on function private.enforce_post_publication_fields() from public;

create trigger posts_enforce_publication_fields
before insert or update on public.posts
for each row execute function private.enforce_post_publication_fields();
```

所有具有 `updated_at` 的表使用统一触发器更新该字段。文章软删除将 `deleted_at` 设为当前时间并把状态改为 `archived`；V1.0.0 不建设回收站 UI 和自动清理任务。

## 7. RLS 与访问策略

所有 `public` schema 业务表必须在创建后立即启用 RLS。授权分两层：Postgres `GRANT` 决定角色是否能访问表，RLS policy 决定具体能访问哪些行。

### 角色判断

所有管理员写策略使用同一判断逻辑：

```sql
exists (
  select 1
  from public.profiles
  where profiles.id = (select auth.uid())
    and profiles.role = 'owner'
)
```

不要使用可被用户编辑的 `user_metadata` 存角色，也不要只写 `TO authenticated`；后者只说明“已登录”，并不说明该用户有管理权限。

### 策略矩阵

| 资源 | `anon` | 普通已登录用户 | `owner` |
| --- | --- | --- | --- |
| `profiles` | 无权限 | 仅能读自己的记录 | 仅能读自己的记录 |
| `posts` | 仅读取已发布、发布时间已到、且未删除的文章 | 与匿名访客相同 | 可读取、创建、更新和软删除文章；不可物理删除。 |
| `categories`、`tags`、`post_tags` | 仅读取公开页面所需关系 | 与匿名访客相同 | 可管理全部内容 |
| `media` | 仅读取已公开文章的封面媒体元数据 | 与匿名访客相同 | 可读取、创建、更新媒体记录；不可物理删除。 |
| `storage.objects` | 公开读取 `post-images` 的文件 URL | 无写入权限 | V1.0.0 仅可上传；不提供物理删除 |

文章公开读取策略必须等价于：

```sql
status = 'published'
and published_at <= now()
and deleted_at is null
```

### RLS migration 的实际规则

策略矩阵必须落实到 `supabase/migrations` 中的逐表 SQL，而不是停留在文档描述。建议拆分为 `001_core_schema.sql` 与 `002_rls_and_storage.sql`；后者对每张表显式执行 `GRANT`、`ENABLE ROW LEVEL SECURITY` 和 `CREATE POLICY`。新建 Supabase 项目中的表不会再自动暴露给 Data API，因此显式 `GRANT` 是 migration 的必要部分；迁移完成后必须分别用匿名和 owner 会话执行 Data API 冒烟测试。

```sql
-- 所有业务表都启用 RLS；以下以 posts 为例。
grant select on public.posts to anon, authenticated;
grant insert, update on public.posts to authenticated;
revoke delete on public.posts from anon, authenticated;

alter table public.posts enable row level security;

create policy "public reads visible posts"
on public.posts for select to anon, authenticated
using (
  status = 'published'
  and published_at <= now()
  and deleted_at is null
);

create policy "owner reads every post"
on public.posts for select to authenticated
using (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);

create policy "owner creates posts"
on public.posts for insert to authenticated
with check (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
  and status <> 'published'
);

create policy "owner updates posts"
on public.posts for update to authenticated
using (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
)
with check (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
  and status <> 'published'
);
```

`posts` 不创建 `DELETE` policy；文章删除只能通过 `UPDATE posts SET deleted_at = now(), status = 'archived'` 实现。CMS 的直接更新策略明确拒绝新状态为 `published` 的行，因此文章发布不会绕过 Nitro；如需修改已发布文章，先将其转回草稿，编辑后再重新发布。`categories`、`tags` 与 `post_tags` 同样只授权其当前版本所需的操作，避免使用笼统的 `FOR ALL`。

`categories`、`tags`、`post_tags` 的公开策略不能照抄文章条件，因为它们没有 `status` 或 `deleted_at` 字段；必须通过关联的可见文章判断。例如：

```sql
-- categories：至少关联一篇可见文章。
exists (
  select 1 from public.posts
  where posts.category_id = categories.id
    and posts.status = 'published'
    and posts.published_at <= now()
    and posts.deleted_at is null
)

-- tags：至少关联一篇可见文章。
exists (
  select 1
  from public.post_tags
  join public.posts on posts.id = post_tags.post_id
  where post_tags.tag_id = tags.id
    and posts.status = 'published'
    and posts.published_at <= now()
    and posts.deleted_at is null
)
```

`post_tags` 的公开读取策略同样通过其 `post_id` 关联可见文章。`media` 需要保留受限的公开读取策略：仅当该媒体被一篇可见文章作为 `cover_media_id` 引用时，才可读取它的 `bucket` 和 `path`。否则使用 publishable key 的前台无法从 `cover_media_id` 生成封面图片 URL。

```sql
grant select on public.media to anon, authenticated;
grant insert, update on public.media to authenticated;
revoke delete on public.media from anon, authenticated;

alter table public.media enable row level security;

create policy "public reads media used as visible cover"
on public.media for select to anon, authenticated
using (
  exists (
    select 1 from public.posts
    where posts.cover_media_id = media.id
      and posts.status = 'published'
      and posts.published_at <= now()
      and posts.deleted_at is null
  )
);

create policy "owner reads all media"
on public.media for select to authenticated
using (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);

create policy "owner inserts media"
on public.media for insert to authenticated
with check (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);

create policy "owner updates media"
on public.media for update to authenticated
using (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
)
with check (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);
```

`profiles` 只授予 `authenticated` 的 `SELECT`，并且只允许 `id = auth.uid()`；它不向浏览器开放 `INSERT`、`UPDATE`、`DELETE` policy。所有 `UPDATE` policy 都必须同时声明 `USING` 与 `WITH CHECK`。

`002_rls_and_storage.sql` 应至少覆盖以下逐表规则：

| 表 | `GRANT` | policy |
| --- | --- | --- |
| `profiles` | `authenticated` 仅 `SELECT` | `id = auth.uid()` 的自读策略；无浏览器写策略。 |
| `posts` | `anon`、`authenticated` 可 `SELECT`；`authenticated` 可 `INSERT`、`UPDATE` | 公开读取、owner 全量读取、owner 插入、owner 更新；无 `DELETE`。 |
| `categories`、`tags`、`post_tags` | 公开读取所需 `SELECT`；只授予 owner 需要的写入操作 | 公开策略通过可见文章关联判断；owner 策略按操作拆分。 |
| `media` | `anon`、`authenticated` 可 `SELECT`；`authenticated` 可 `INSERT`、`UPDATE` | 仅可读可见文章的封面媒体；owner 可读、插入、更新；无删除策略。 |
| `storage.objects` | Storage 内置表，不直接修改表权限 | `post-images` 的 owner `INSERT` 策略；V1.0.0 无 `DELETE`、`UPDATE` policy。 |

## 8. 媒体与图片

创建一个名为 `post-images` 的 **公开 bucket**，只存文章正文和封面需要公开展示的图片。

### 上传规则

- CMS 使用随机 UUID 生成路径，例如 `posts/<post-id>/<uuid>.webp`。
- 不使用 Storage upsert；同名覆盖会增加额外 `SELECT` 与 `UPDATE` 权限需求。
- bucket 设置 MIME 白名单（JPEG、PNG、WebP、GIF）和单文件大小上限。
- 浏览器上传前可以压缩图片，但服务器权限不依赖前端校验。
- `media` 保存 `bucket`、`path`、`mime_type`、`size_bytes`、`alt` 和上传时间；文章保存 `cover_media_id`，而不是永久写死完整 URL。
- V1.0.0 不提供“彻底删除图片”。正文 Markdown 可能仍引用旧图；后续先实现引用检查，再同时删除 `media` 记录和 Storage 对象。

`storage.objects` 的上传 policy 必须同时限制 bucket 和 owner 身份。V1.0.0 只创建 `INSERT` policy，不创建 `DELETE` 或 upsert 所需的 `SELECT`、`UPDATE` policy：

```sql
create policy "owner uploads post images"
on storage.objects for insert to authenticated
with check (
  bucket_id = 'post-images'
  and exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);
```

图片 bucket 是公开的，意味着任何拿到 URL 的人都可读取图片；它不适用于草稿附件、备份或私密文件。私有文件需求出现后，再单独创建私有 bucket 与签名 URL 流程。

由于 V1.0.0 只使用公开 bucket，**图片在上传完成时即已公开**，即使文章仍是草稿。随机路径只降低偶然猜测的概率，不构成访问控制。因此后台不得上传敏感、未授权或不希望提前公开的图片；若未来需要真正私密的草稿媒体，再增加私有 bucket 与发布时复制文件的流程。

## 9. 前台与 CMS 的职责

### `apps/blog-web`

- Nuxt SSR 页面：首页、文章列表、文章详情、分类、标签、归档、关于页。
- 通过 Supabase publishable key 读取数据，依赖 RLS 过滤草稿和删除内容。
- 渲染受限 Markdown、代码高亮、Canonical、Open Graph、RSS、Sitemap。
- 只提供一个管理写接口：`POST /api/admin/posts/:id/publish`。
- 项目展示、友情链接、导航与站点基础配置在 V1.0.0 由前台代码静态维护，不建立对应后台表和 CMS 页面。

发布接口先使用 CMS 传来的 Supabase access token 调用 `auth.getUser()`，再用该用户态 client 读取自己的 `profiles.role`，确认其为 owner。验证通过后，Nitro 才在服务端使用私有 `SUPABASE_SERVICE_ROLE_KEY` 将指定文章切换为 `published`；该 key 绝不进入浏览器或公共运行时配置。发布时间仍由数据库触发器写入。

V1.0.0 不存在 `scheduled_at`、Cron 或后台队列；发布只由管理员手动触发。AI、邮件订阅、数据导出、浏览量统计、登录日志与操作日志也不进入当前应用或数据库模型。

### Nitro 发布接口的跨域约束

CMS 部署在 `https://admin.example.com`，发布接口位于 `https://www.example.com`，因此该请求是跨域请求。发布接口只接受以下浏览器跨域规则：

```text
Origin: https://admin.example.com
Method: POST
Allowed request headers: Authorization, Content-Type
```

接口必须显式处理 `OPTIONS` 预检请求。对于 `OPTIONS` 和 `POST`，仅当 `Origin` 与环境变量 `NUXT_CMS_ORIGIN` 完全相等时才继续处理，并返回 `Access-Control-Allow-Origin`、`Access-Control-Allow-Methods: POST` 与 `Access-Control-Allow-Headers: Authorization, Content-Type`。不得使用 `Access-Control-Allow-Origin: *`，也不得在 `nuxt.config.ts` 中对 `/api/**` 设置全站 `cors: true`。

CORS 只限制浏览器是否允许跨站读取响应，不是鉴权机制。`POST /api/admin/posts/:id/publish` 必须先验证 Supabase access token 和该用户的 `profiles.role = 'owner'`，验证失败时在使用 service key 前返回 `401` 或 `403`；RLS 则负责阻止浏览器绕过 Nitro 直接将文章状态改为 `published`。

实现时，`publish.post.ts` 和 `publish.options.ts` 复用 `server/utils/admin-cors.ts`：允许的预检请求返回 `204`，携带上述三个允许头以及 `Vary: Origin`；不匹配的 Origin 直接返回 `403`。验收应覆盖允许 Origin 的 `OPTIONS`、未知 Origin 的 `OPTIONS`、无 token 的 `POST` 和非 owner token 的 `POST`。

### `apps/cms-admin`

- 仅管理员可登录并进入路由。
- 草稿保存、分类、标签、媒体记录等简单 CRUD 直接调用 Supabase。
- 图片仅可上传和选择，受 `storage.objects` 的 owner policy 约束；V1.0.0 不提供物理删除。
- 发布按钮只调用 Nitro 发布接口；不允许编辑 `published_at`。
- 前端路由守卫只是体验层，数据库 RLS 才是真正权限控制。
- CMS 不提供公开注册、邀请、用户管理、角色管理、菜单权限或多租户能力。

## 10. Markdown / MDC 规则

- 数据库存储 Markdown/MDC 源文本，而不是 HTML 或编辑器私有 JSON。
- 原始 HTML 默认禁用；禁止 `javascript:` 等危险链接协议。
- 正式文章与草稿最终预览统一在 `blog-web` 使用 `@nuxtjs/mdc` 渲染；CMS 使用 CodeMirror 编辑，不承担最终渲染职责。
- V1.0.1 只支持普通 Markdown、代码块、图片和 content-rules 白名单中的受控 MDC 块；不开放任意 Vue 组件注入。
- Markdown 正文中的图片引用公开 Storage URL 或 `media` 记录生成的 URL。

V1.0.0 不创建 PostgreSQL 全文搜索索引或接入第三方搜索服务。文章数量较少时，可在 CMS 中筛选标题；访客端搜索在后续通过中文分词方案验证后再加入。

## 11. 缓存、渲染与部署

### Nuxt 缓存规则

```ts
export default defineNuxtConfig({
  routeRules: {
    '/': { isr: 600 },
    '/posts': { isr: 600 },
    '/posts/**': { isr: 600 },
    '/categories/**': { isr: 600 },
    '/tags/**': { isr: 600 },
    '/api/**': { cache: false },
  },
})
```

内容页部署在 Netlify 时使用 10 分钟 ISR，让 Netlify CDN 成为唯一的 HTML 缓存层；CMS 页面和写入 API 不缓存。发布后最多 10 分钟看到旧内容是 V1.0.0 的明确产品行为，不是故障。

### 部署约定

- `blog-web` 部署为 Netlify 上支持 SSR/Nitro 的 Nuxt 应用。
- `cms-admin` 部署为独立 Netlify 静态站点。
- `www.example.com` 指向博客，`admin.example.com` 指向 CMS。
- 若使用 Cloudflare，只托管 DNS；不要开启代理缓存，避免 Netlify、Nitro、Cloudflare 三层同时缓存 HTML。
- `cms-admin` 的前端环境变量仅包含 Supabase URL 与 publishable key。`blog-web` 的 Nitro 私有运行时配置额外保存 `SUPABASE_SERVICE_ROLE_KEY`，仅供已完成身份校验的发布接口使用；任何 secret 都不进入浏览器构建产物或 `public` runtime config。
- 部署后用浏览器 Network 面板或 `curl -I` 检查内容页缓存响应头，确认不存在超过 600 秒的额外 HTML 缓存。

## 12. 开发阶段与验收标准

### 阶段 1：项目与数据库基础

1. 初始化 pnpm workspace、Nuxt、Vue CMS 与 Supabase CLI。
2. 创建核心表、索引、约束和时间戳触发器。
3. 创建 RLS、Storage policy 与 `post-images` bucket。
4. 关闭公开注册，并创建唯一 owner。
5. 用 publishable key 分别以匿名与 owner 身份执行 Data API 冒烟测试，确认显式 `GRANT` 已使所需表可访问且 RLS 结果正确。

**验收：** 匿名用户和非 owner 登录用户都只能读取公开内容，无法读取草稿或已删除文章；owner 可以完成当前版本允许的内容管理操作；任何身份都不能直接 `DELETE posts`。

### 阶段 2：CMS 内容工作流

1. 登录、登出与修改密码。
2. 文章草稿编辑、分类标签管理。
3. 图片上传、选图、封面选择与文章软删除。
4. Nitro 发布接口与发布按钮。

**验收：** 管理员可从空白草稿完成“上传图片 → 编辑 Markdown → 发布”流程，且发布时间不可由 CMS 指定为未来时间；浏览器直连 Data API 无法将状态改为 `published`；允许 Origin 的 `OPTIONS` 返回 `204`，未知 Origin 返回 `403`，无 token 与非 owner token 均不能发布。

### 阶段 3：公开博客

1. 首页、列表、详情、分类、标签、归档。
2. 统一 Markdown 渲染、SEO、RSS 与 Sitemap。
3. 配置 10 分钟 ISR 并部署预览环境。

**验收：** 未发布或软删除文章无论通过页面、接口还是直接调用 Data API 均不可访问；发布内容在最长 10 分钟内出现在前台；公开文章的 `cover_media_id` 可解析为图片 URL，未被公开文章引用的媒体元数据不可被匿名读取。

### 阶段 4：上线检查

1. 检查浏览器构建产物中没有 secret key。
2. 用匿名、普通已登录、owner 三种身份执行权限测试。
3. 检查 Storage 只能由 owner 上传；确认 V1.0.0 没有物理删除图片的接口或 policy。
4. 检查响应缓存头，确认没有超过 10 分钟的额外 HTML 缓存。
5. 配置生产域名、Supabase Auth Site URL 与允许的 redirect URL。
6. 执行 `supabase db advisors`，修复 RLS、函数或暴露 schema 的安全告警。

## 13. 后续演进顺序

V1.0.0 稳定后，按以下顺序增加能力：

1. 草稿预览（管理员鉴权 + `noindex`）。
2. 项目、友链、站点配置的后台管理。
3. 中文搜索方案验证与搜索索引。
4. 操作日志、导出和备份。
5. 发布 webhook 与精确缓存失效。
6. 定时发布与 Cron。
7. AI、邮件订阅及其他第三方服务。

任何新表都必须先补齐 RLS、索引、约束和测试，再接入 CMS 页面。
