# 个人技术博客 + 独立 CMS 技术方案 V1.0.13

## 1. 项目概述

本项目是一个单管理员维护的个人技术博客，包含一个面向访客的博客前台和一个独立的内容管理后台（CMS）。

V1.0.13 是 V1.0.0 产品范围的技术方案修订版，保持安全、稳定的内容发布链路，并把 CMS 从可操作原型提升为完整的单管理员内容工作台：管理员可以以邮箱密码登录 CMS，通过真实概览、可筛选分页的文章列表、结构化编辑器、分类标签和媒体工作区管理内容；访客可以浏览文章列表、文章详情、分类、标签、归档与关于页面。博客前台提供 SSR、响应式图片、页面元信息、摘要型 RSS 和 Sitemap；Netlify CDN 严格缓存 600 秒，并按主题 Cookie 隔离缓存对象，不继续提供过期内容。本修订将公开博客的读取链路统一收口到同源 Nitro API：浏览器不再直接调用 Supabase Data API，Nitro 使用私有运行时中的 publishable key 并继续受 RLS 约束，同时输出稳定的公开 DTO 并复用 Netlify 600 秒缓存规则。CMS 普通管理仍直接使用 Supabase，以保留简单、可验证的单管理员工作流。

项目采用渐进式建设。未进入当前版本的功能会在相应技术章节说明，不单独建设一套“待办功能清单”。

## 2. 总体架构

```text
访客 ──→ blog-web（Nuxt 4 页面）──→ Nitro 公开读取 API ──┐
       SSR / 客户端导航             GET /api/public/**   │ publishable key
       SEO / RSS / Sitemap          Netlify CDN 600 秒    ▼
                                                   ┌───────────────┐
                                                   │ Supabase      │
                                                   │ PostgreSQL    │
                                                   │ RLS 最终边界   │
                                                   └───────▲───────┘
                                                           │ 管理员 JWT
管理员 ──→ cms-admin（Vue 3 SPA）───────────────────────────┤
          草稿 CRUD / Auth / Storage 直接调用 Supabase      │
          │                                                │
          └──→ Nitro 发布 API ──→ 校验 token + owner ──────┘
               POST /api/admin/posts/:postId/publish    server secret
```

### 架构原则

1. **Supabase 是唯一在线数据源**：文章 Markdown、分类、标签和媒体记录都在 Supabase；不同时维护 `content/*.md` 作为第二份主数据。
2. **RLS 是安全边界**：Nitro 公开读取与 CMS 直接调用都不能绕过 RLS；公开读取只使用 publishable key，数据库必须拒绝未发布、未来发布或软删除内容。
3. **按信任边界使用 Nitro**：公开博客页面只调用同源 `GET /api/public/**`，由 Nitro 查询 Supabase 并返回稳定 DTO；发布等特权状态迁移也经过 Nitro。CMS 普通草稿 CRUD、Auth 与 Storage 继续直接调用 Supabase，避免重复建设管理端 BFF。
4. **浏览器永不持有服务端密钥**：公开博客的 publishable key 也只存在于 Nitro 私有运行时；CMS 浏览器只使用低权限 publishable key 与用户 JWT。`service_role`、邮件或 AI 密钥只允许可信服务端使用，V1.0.0 中 server secret 仅由 Nitro 发布接口在完成 token 与 owner 校验后使用。
5. **缓存可预测优先于即时生效**：V1.0.0 只接受 10 分钟缓存延迟，不增加 webhook 刷新或第二层 CDN 缓存。

### 公开读取接口

| 接口 | 响应边界 |
| --- | --- |
| `GET /api/public/posts` | 已发布文章列表 `PostPreview[]`。 |
| `GET /api/public/posts/:postSlug` | 单篇 `PostDetail` 与上一篇/下一篇导航；不可见或不存在时返回 404。 |
| `GET /api/public/categories` | 至少被一篇可见文章引用的分类。 |
| `GET /api/public/tags` | 至少被一篇可见文章引用的标签。 |

这些接口不接收 Supabase token，不使用 server secret，也不自行复制发布状态条件；它们使用 publishable key 访问 Data API，由数据库 `GRANT + RLS` 决定可见行。Nitro 负责隐藏数据库列命名和关联结构、生成共享 DTO、统一错误语义与 CDN 缓存。RSS 和 Sitemap 复用同一服务端公开查询层，不各自创建不同权限的客户端。

### 远程 Supabase 工作流

本项目只连接专用远程 Supabase 项目，不启动本地 Supabase 或 Docker。schema 变更仍必须先用 `supabase migration new` 创建迁移文件，再通过已链接的远程项目执行 `pnpm db:push`；类型使用 `pnpm db:types` 从远程 schema 生成。`pnpm db:test` 通过 Supabase Management API 的 `db query --linked` 在该专用项目执行 pgTAP，测试必须使用事务回滚，不能指向任何共享或生产数据库。

## 5. 身份与管理员初始化

V1.0.0 只有一个角色：`owner`。

### 必须遵守的初始化流程

1. 在 Supabase Auth 设置中关闭公开注册。
2. 在 Supabase Dashboard 手动创建唯一的邮箱密码用户。
3. 使用受控 SQL 为该用户的 UID 插入一条 `profiles` 记录，角色为 `owner`。
4. CMS 只提供登录、登出和修改当前密码；不出现注册、邀请、角色管理或用户管理页面。
5. 生产 Auth Site URL 设为 CMS 的精确 HTTPS 地址，需要邮件重定向时只增加明确的允许 URL。
6. 前台和 CMS 使用现代 `sb_publishable_...` key，Nitro 私有运行时使用 `sb_secret_...` key；确认部署无误后禁用 legacy JWT API keys。
7. Supabase Auth 与 CMS 都要求密码至少 8 个字符；远程配置与 `supabase/config.toml` 必须保持一致。

`profiles.role` **不能有 `owner` 默认值**，也不能由浏览器插入或修改。仅在管理端创建账号时写入一次。

```sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  display_name varchar(100) not null default '',
  role varchar(20) not null check (role in ('owner')),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- 在 Dashboard 创建 Auth 用户后，以数据库管理权限执行：
insert into public.profiles (id, display_name, role)
values ('<AUTH_USER_UUID>', 'Chen', 'owner');
```

## 6. 数据模型

V1.0.0 只建立以下核心表。

| 表 | 作用 |
| --- | --- |
| `profiles` | 唯一管理员的身份与角色。 |
| `posts` | Markdown 文章、发布状态和软删除状态。 |
| `categories` | 文章分类。 |
| `tags` | 文章标签。 |
| `post_tags` | 文章与标签的多对多关系。 |
| `media` | Storage 文件的元数据和引用目标。 |

迁移的建表顺序为：`profiles` → `categories`、`tags` → `media` → `posts` → `post_tags`，以满足外键依赖。

### `posts` 的关键字段与约束

```sql
create table public.posts (
  id uuid primary key default gen_random_uuid(),
  title varchar(200) not null,
  slug varchar(200) not null unique,
  summary text not null default '',
  content text not null default '',
  cover_media_id uuid references public.media(id) on delete set null,
  category_id uuid references public.categories(id) on delete restrict,
  status varchar(20) not null default 'draft'
    check (status in ('draft', 'published', 'archived')),
  published_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  deleted_at timestamptz,
  constraint published_posts_have_timestamp
    check (status <> 'published' or published_at is not null)
);

create index posts_public_listing_idx
  on public.posts (published_at desc)
  where status = 'published' and deleted_at is null;
```

不创建 `scheduled_at`。发布时间由数据库触发器强制维护：新建的已发布文章或状态首次切换为 `published` 时，触发器将 `published_at` 写为 `now()`；已发布文章不得单独修改该字段；文章重新归档或转回草稿时将其清空，下一次发布再次写入当前时间。因此客户端无法将文章设置为未来发布时间。

```sql
create schema if not exists private;

create function private.enforce_post_publication_fields()
returns trigger
language plpgsql
set search_path = ''
as $$
begin
  if new.status = 'published' then
    if tg_op = 'INSERT' then
      new.published_at := now();
    elsif old.status is distinct from 'published' then
      new.published_at := now();
    elsif new.published_at is distinct from old.published_at then
      raise exception 'published_at cannot be edited directly';
    end if;
  else
    new.published_at := null;
  end if;

  return new;
end;
$$;

revoke all on function private.enforce_post_publication_fields() from public;

create trigger posts_enforce_publication_fields
before insert or update on public.posts
for each row execute function private.enforce_post_publication_fields();
```

所有具有 `updated_at` 的表使用统一触发器更新该字段。文章软删除将 `deleted_at` 设为当前时间并把状态改为 `archived`；V1.0.0 不建设回收站 UI 和自动清理任务。

## 7. RLS 与访问策略

所有 `public` schema 业务表必须在创建后立即启用 RLS。授权分两层：Postgres `GRANT` 决定角色是否能访问表，RLS policy 决定具体能访问哪些行。

### 角色判断

所有管理员写策略使用同一判断逻辑：

```sql
exists (
  select 1
  from public.profiles
  where profiles.id = (select auth.uid())
    and profiles.role = 'owner'
)
```

不要使用可被用户编辑的 `user_metadata` 存角色，也不要只写 `TO authenticated`；后者只说明“已登录”，并不说明该用户有管理权限。

### 策略矩阵

| 资源 | `anon` | 普通已登录用户 | `owner` |
| --- | --- | --- | --- |
| `profiles` | 无权限 | 仅能读自己的记录 | 仅能读自己的记录 |
| `posts` | 仅读取已发布、发布时间已到、且未删除的文章 | 与匿名访客相同 | 可读取、创建、更新和软删除文章；不可物理删除。 |
| `categories`、`tags`、`post_tags` | 仅读取公开页面所需关系 | 与匿名访客相同 | 可创建并管理草稿所需关系；已发布文章的分类、标签元数据和关系不可直接改动，被任何文章引用的分类或标签不可删除。 |
| `media` | 仅读取已公开文章的封面媒体元数据 | 与匿名访客相同 | 可读取、创建、更新媒体记录；已发布文章正在使用的封面元数据需先转草稿再改，不可物理删除。 |
| `storage.objects` | 公开读取 `post-images` 的文件 URL | 无写入权限 | V1.0.0 仅可上传；不提供物理删除 |

文章公开读取策略必须等价于：

```sql
status = 'published'
and published_at <= now()
and deleted_at is null
```

### RLS migration 的实际规则

策略矩阵必须落实到 `supabase/migrations` 中的逐表 SQL，而不是停留在文档描述。建议拆分为 `001_core_schema.sql` 与 `002_rls_and_storage.sql`；后者对每张表显式执行 `GRANT`、`ENABLE ROW LEVEL SECURITY` 和 `CREATE POLICY`。新建 Supabase 项目中的表不会再自动暴露给 Data API，因此显式 `GRANT` 是 migration 的必要部分；迁移完成后必须分别用匿名和 owner 会话执行 Data API 冒烟测试。

```sql
-- 所有业务表都启用 RLS；以下以 posts 为例。
grant select on public.posts to anon, authenticated;
grant insert, update on public.posts to authenticated;
revoke delete on public.posts from anon, authenticated;

alter table public.posts enable row level security;

create policy "public reads visible posts"
on public.posts for select to anon, authenticated
using (
  status = 'published'
  and published_at <= now()
  and deleted_at is null
);

create policy "owner reads every post"
on public.posts for select to authenticated
using (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);

create policy "owner creates posts"
on public.posts for insert to authenticated
with check (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
  and status <> 'published'
);

create policy "owner updates posts"
on public.posts for update to authenticated
using (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
)
with check (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
  and status <> 'published'
);
```

`posts` 不创建 `DELETE` policy；文章删除只能通过 `UPDATE posts SET deleted_at = now(), status = 'archived'` 实现。CMS 的直接更新策略明确拒绝新状态为 `published` 的行，因此文章发布不会绕过 Nitro；如需修改已发布文章，先将其转回草稿，编辑后再重新发布。`categories`、`tags` 与 `post_tags` 同样只授权其当前版本所需的操作，避免使用笼统的 `FOR ALL`。

已发布文章是一个包含文章主体、分类、标签关系和封面替代文本的公开快照。除 `posts` 更新策略外，migration 还要使用 `AS RESTRICTIVE` 写策略：`post_tags` 只能为未删除的草稿增删关系，被已发布文章引用的分类、标签与封面媒体不允许更新。`post_tags` 不授予 `UPDATE`；关系更换使用受控的删除和插入。

PostgreSQL 外键的 referential action 可能以表 owner 权限执行，不能仅依赖子表 RLS 阻止间接改动。因此 `posts.category_id` 与 `post_tags.tag_id` 都使用 `ON DELETE RESTRICT`；被任何文章引用的分类或标签必须先解除关系才能删除。为支持全状态外键检查，另建立非部分索引 `posts(category_id)`。

`categories`、`tags`、`post_tags` 的公开策略不能照抄文章条件，因为它们没有 `status` 或 `deleted_at` 字段；必须通过关联的可见文章判断。例如：

```sql
-- categories：至少关联一篇可见文章。
exists (
  select 1 from public.posts
  where posts.category_id = categories.id
    and posts.status = 'published'
    and posts.published_at <= now()
    and posts.deleted_at is null
)

-- tags：至少关联一篇可见文章。
exists (
  select 1
  from public.post_tags
  join public.posts on posts.id = post_tags.post_id
  where post_tags.tag_id = tags.id
    and posts.status = 'published'
    and posts.published_at <= now()
    and posts.deleted_at is null
)
```

`post_tags` 的公开读取策略同样通过其 `post_id` 关联可见文章。`media` 需要保留受限的公开读取策略：仅当该媒体被一篇可见文章作为 `cover_media_id` 引用时，才可读取它的 `bucket_id` 和 `object_path`。否则使用 publishable key 的前台无法从 `cover_media_id` 生成封面图片 URL。

```sql
grant select on public.media to anon, authenticated;
grant insert, update on public.media to authenticated;
revoke delete on public.media from anon, authenticated;

alter table public.media enable row level security;

create policy "public reads media used as visible cover"
on public.media for select to anon, authenticated
using (
  exists (
    select 1 from public.posts
    where posts.cover_media_id = media.id
      and posts.status = 'published'
      and posts.published_at <= now()
      and posts.deleted_at is null
  )
);

create policy "owner reads all media"
on public.media for select to authenticated
using (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);

create policy "owner inserts media"
on public.media for insert to authenticated
with check (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);

create policy "owner updates media"
on public.media for update to authenticated
using (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
)
with check (
  exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);
```

`profiles` 只授予 `authenticated` 的 `SELECT`，并且只允许 `id = auth.uid()`；它不向浏览器开放 `INSERT`、`UPDATE`、`DELETE` policy。所有 `UPDATE` policy 都必须同时声明 `USING` 与 `WITH CHECK`。

`002_rls_and_storage.sql` 应至少覆盖以下逐表规则：

| 表 | `GRANT` | policy |
| --- | --- | --- |
| `profiles` | `authenticated` 仅 `SELECT` | `id = auth.uid()` 的自读策略；无浏览器写策略。 |
| `posts` | `anon`、`authenticated` 可 `SELECT`；`authenticated` 可 `INSERT`、`UPDATE` | 公开读取、owner 全量读取、owner 插入、owner 更新；无 `DELETE`。 |
| `categories`、`tags`、`post_tags` | 公开读取所需 `SELECT`；`post_tags` 仅授予 `INSERT`、`DELETE`，不授予 `UPDATE` | 公开策略通过可见文章关联判断；owner permissive 策略按操作拆分，再与发布快照的 restrictive 策略组合。 |
| `media` | `anon`、`authenticated` 可 `SELECT`；`authenticated` 可 `INSERT`、`UPDATE` | 仅可读可见文章的封面媒体；owner 可读、插入、更新未被已发布文章作为封面使用的记录；无删除策略。 |
| `storage.objects` | Storage 内置表，不直接修改表权限 | `post-images` 的 owner `INSERT` 策略；V1.0.0 无 `DELETE`、`UPDATE` policy。 |

## 8. 媒体与图片

创建一个名为 `post-images` 的 **公开 bucket**，只存文章正文和封面需要公开展示的图片。

### 展示与优化规则

- `blog-web` 使用 `@nuxt/image` v2；文章封面直接使用 `<NuxtImg>`，Comark 正文图片通过遵循 Nuxt UI prose 约定的自定义 `ProseImg` 组件交给 `<NuxtImg>`，统一输出懒加载、异步解码和响应式 `sizes`。
- 本地开发使用默认 IPX；部署到 Netlify 后让模块自动选择 Netlify Image CDN，不在代码中强制指定 `netlify` provider。
- 构建时从 `NUXT_PUBLIC_SUPABASE_URL` 提取并精确加入 `image.domains`。Netlify 构建环境必须提供该变量，不允许泛化为任意 Supabase 或外部域名。
- 当前不使用 Nuxt Image 的 `supabase` provider，因为 Supabase Image Transformations 需要 Pro 套餐。未来升级套餐后再单独评估，V1 不同时叠加两套图片转换服务。
- `content-rules` 将普通链接协议与图片来源分开校验；Markdown 图片只能指向当前项目 `/storage/v1/object/public/post-images/` 路径。CMS 保存和 Nitro 发布必须使用同一前缀规则。
- 封面输出显式宽高、响应式 `sizes`、替代文本和首屏加载优先级，避免布局偏移；CMS 是纯 Vue/Vite 应用，不安装 `@nuxt/image`。

### 上传规则

- CMS 在 bucket 根目录使用随机 UUID 生成对象路径，例如 `<uuid>.webp`；扩展名按实际 MIME 类型映射，当前不建立按文章 ID 分层的子目录。
- 不使用 Storage upsert；同名覆盖会增加额外 `SELECT` 与 `UPDATE` 权限需求。
- bucket 设置 MIME 白名单（JPEG、PNG、WebP、GIF）和单文件大小上限。
- 浏览器上传前可以压缩图片，但服务器权限不依赖前端校验。
- `media` 保存 `id`、`bucket_id`、`object_path`、`alt_text`、`mime_type`、`size_bytes`、`created_at` 和 `updated_at`；文章保存 `cover_media_id`，而不是永久写死完整 URL。
- V1.0.0 不提供“彻底删除图片”。正文 Markdown 可能仍引用旧图；后续先实现引用检查，再同时删除 `media` 记录和 Storage 对象。

当前上传是两个顺序请求：先将对象写入 Storage，再插入 `media` 记录，两者不具备跨服务事务性。如果 Storage 上传已成功而 `media` 插入失败，公开 bucket 会留下没有元数据记录的孤儿对象；V1.0.0 没有 `storage.objects` 的 `DELETE` policy，浏览器无法做补偿删除。运营上应将这类失败视为需人工核对的异常：停止盲目重试，在 Supabase Dashboard 或其他受控运维通道核对并必要时清理最近对象。CMS 不增加删除入口或特权补偿 API，此限制按 V1 边界接受。

`storage.objects` 的上传 policy 必须同时限制 bucket 和 owner 身份。V1.0.0 只创建 `INSERT` policy，不创建 `DELETE` 或 upsert 所需的 `SELECT`、`UPDATE` policy：

```sql
create policy "owner uploads post images"
on storage.objects for insert to authenticated
with check (
  bucket_id = 'post-images'
  and exists (
    select 1 from public.profiles
    where profiles.id = (select auth.uid())
      and profiles.role = 'owner'
  )
);
```

图片 bucket 是公开的，意味着任何拿到 URL 的人都可读取图片；它不适用于草稿附件、备份或私密文件。私有文件需求出现后，再单独创建私有 bucket 与签名 URL 流程。

由于 V1.0.0 只使用公开 bucket，**图片在上传完成时即已公开**，即使文章仍是草稿。随机路径只降低偶然猜测的概率，不构成访问控制。因此后台不得上传敏感、未授权或不希望提前公开的图片；若未来需要真正私密的草稿媒体，再增加私有 bucket 与发布时复制文件的流程。

## 9. 前台与 CMS 的职责

### `apps/blog-web`

- Nuxt SSR 页面：首页、文章列表、文章详情、分类、标签、归档、关于页。
- 通过 Supabase publishable key 读取数据，依赖 RLS 过滤草稿和删除内容。
- 渲染受限 Markdown、可横向滚动的代码块、h2–h4 文章目录、Canonical、Open Graph、摘要型 RSS、Sitemap。
- 只提供一个管理写接口：`POST /api/admin/posts/:id/publish`。
- 项目展示、友情链接、导航与站点基础配置在 V1.0.0 由前台代码静态维护，不建立对应后台表和 CMS 页面。

发布接口先使用 CMS 传来的 Supabase access token 调用 `auth.getUser()`，再用该用户态 client 读取自己的 `profiles.role`，确认其为 owner。验证通过后，Nitro 才在服务端使用私有运行时变量 `NUXT_SUPABASE_SERVICE_ROLE_KEY` 将指定文章切换为 `published`；该 key 绝不进入浏览器或公共运行时配置。发布时间仍由数据库触发器写入。

Nitro 读取待发布草稿时必须同时读取 `updated_at`。标题、Slug、正文与 Markdown 校验通过后，最终更新除了限制 `id`、`status = 'draft'` 和 `deleted_at is null`，还必须限制 `updated_at` 与已校验版本完全一致；若校验与更新之间草稿发生变化，则返回 `409` 并要求重试，不得发布未经本次校验的新版本。

V1.0.0 不存在 `scheduled_at`、Cron 或后台队列；发布只由管理员手动触发。AI、邮件订阅、数据导出、浏览量统计、登录日志与操作日志也不进入当前应用或数据库模型。

### Nitro 发布接口的跨域约束

CMS 部署在 `https://admin.example.com`，发布接口位于 `https://www.example.com`，因此该请求是跨域请求。发布接口只接受以下浏览器跨域规则：

```text
Origin: https://admin.example.com
Method: POST
Allowed request headers: Authorization, Content-Type
```

接口必须显式处理 `OPTIONS` 预检请求。对于 `OPTIONS` 和 `POST`，仅当 `Origin` 与环境变量 `NUXT_CMS_ORIGIN` 完全相等时才继续处理，并返回 `Access-Control-Allow-Origin`、`Access-Control-Allow-Methods: POST` 与 `Access-Control-Allow-Headers: Authorization, Content-Type`。不得使用 `Access-Control-Allow-Origin: *`，也不得在 `nuxt.config.ts` 中对 `/api/**` 设置全站 `cors: true`。

CORS 只限制浏览器是否允许跨站读取响应，不是鉴权机制。`POST /api/admin/posts/:id/publish` 必须先验证 Supabase access token 和该用户的 `profiles.role = 'owner'`，验证失败时在使用 service key 前返回 `401` 或 `403`；RLS 则负责阻止浏览器绕过 Nitro 直接将文章状态改为 `published`。

实现时，`publish.post.ts` 和 `publish.options.ts` 复用 `server/utils/admin-cors.ts`：允许的预检请求返回 `204`，携带上述三个允许头以及 `Vary: Origin`；不匹配的 Origin 直接返回 `403`。验收应覆盖允许 Origin 的 `OPTIONS`、未知 Origin 的 `OPTIONS`、无 token 的 `POST` 和非 owner token 的 `POST`。

### `apps/cms-admin`

- 仅管理员可登录并进入路由。
- 草稿保存、分类、标签、媒体记录等简单 CRUD 直接调用 Supabase。
- 图片仅可上传和选择，受 `storage.objects` 的 owner policy 约束；V1.0.0 不提供物理删除。
- 发布按钮只调用 Nitro 发布接口；不允许编辑 `published_at`。
- 前端路由守卫只是体验层，数据库 RLS 才是真正权限控制。
- CMS 不提供公开注册、邀请、用户管理、角色管理、菜单权限或多租户能力。
- CMS 不是公开页面：HTML 设置 `noindex,nofollow,noarchive`，静态 `robots.txt` 全站 `Disallow`，Netlify 响应同时返回 `X-Robots-Tag`。CMS 响应还必须返回 `Content-Security-Policy: frame-ancestors 'none'` 与 `X-Frame-Options: DENY`，避免管理员会话页面被第三方站点嵌入并实施点击劫持。

CMS 的应用壳层继续由项目自研，不导入并裁剪通用后台模板。调研中的 V3 Admin Vite、Pure Admin Thin 与 Vben 均提供了成熟的侧栏、页面容器、搜索表格和状态反馈模式，但同时绑定 Pinia、Axios、动态权限、Mock、国际化、多布局、图表或另一套主题系统；整套迁移会与单 owner、Supabase SDK/RLS 和现有语义 token 形成重复架构。实现只吸收这些项目的信息架构与组件职责，直接复用 Element Plus 原子组件，不复制模板源码，也不增加模板运行时依赖。

CMS 信息架构固定为六个工作区：概览、文章、分类、标签、媒体和账户。应用壳层提供分组侧栏、桌面折叠模式、移动端焦点受控抽屉、当前页面上下文、公开博客入口、快捷新建、主题切换和 owner 菜单。页面层统一使用 `AdminPage`、`DataToolbar`、`AsyncState`、`PaginationBar` 与 `StatusBadge` 等职责明确的组件，所有数据页面必须区分首次载入、保留数据刷新、请求错误、数据库为空、筛选无结果和行级操作中状态。文章列表使用 Supabase `range` 与 exact count 做服务端分页，支持标题搜索、状态和分类筛选、允许列表排序，并将条件同步到 URL；桌面呈现表格，窄屏呈现语义化内容行，不能只依赖横向滚动。

概览只显示真实数据：文章状态数量、分类/标签/媒体数量、最近修改和待处理草稿，不伪造访问量或趋势。当前没有统计数据源，因此 CMS 不使用 Tailwind，不安装 ECharts、`vue-echarts` 或 Yjs；如果后续引入真实统计能力，再独立评估图表方案。

博客前台和 CMS 都支持浅色与暗色主题：默认跟随系统，手动切换后分别持久化，并在应用首屏挂载前应用主题。两端共用设计语义而不共用框架实现，组件不得散落固定主题颜色。

## 10. Markdown / Comark 规则

- 数据库存储 Markdown/Comark 源文本，而不是 HTML 或编辑器私有 JSON。现有 MDC 组件语法与 Comark 兼容，但 V1.0.13 不安装或维护 `@nuxtjs/mdc`。
- `<Comark>` 固定以 `html: false` 渲染，并启用 Comark 官方 `security` 插件及项目严格标签/属性策略。保存与发布前都使用 `content-rules` 的 Comark AST 校验拒绝原始 HTML、危险链接协议、非 `post-images` 图片、未知属性和不在白名单中的组件；渲染端再次执行同一白名单，作为旧数据与校验缺陷的纵深防御。
- 正式发布文章在 `blog-web` 使用 `@comark/nuxt` 渲染；CMS 使用 CodeMirror 编辑并在保存、发布前校验源码。V1 不提供草稿最终预览，避免在 CMS 中维护第二套渲染结果。
- 页面标题是文章唯一的 h1；正文 h1 在正式渲染时降为 h2。正文 h2–h4 使用同一套 Unicode 归一化和重复后缀规则生成稳定 `id`，目录与正文复用该规则；至少有两个目录标题时才显示可键盘访问的目录导航，h5–h6 保留正文层级但不进入目录。
- V1.0.13 只支持普通 Markdown、代码块和图片；Comark 自定义组件白名单初始为空，不开放任意 Vue 组件注入。代码块保持可读与横向滚动，语法高亮作为后续独立能力评估，不在 V1 引入额外插件。
- Markdown 正文中的图片引用公开 Storage URL 或 `media` 记录生成的 URL。

RSS 只输出标题、链接、发布时间与经过 XML 转义的文章摘要，不输出 Markdown 正文，也不生成 `content:encoded`。这样既保持订阅发现能力，也不把代码块中的原始标签传播给下游 RSS 阅读器。公开日期格式化、归档年份和站点展示统一使用 `Asia/Shanghai`。

V1.0.0 不创建 PostgreSQL 全文搜索索引或接入第三方搜索服务。文章数量较少时，可在 CMS 中筛选标题；访客端搜索在后续通过中文分词方案验证后再加入。

## 11. 缓存、渲染与部署

### Netlify 单层 CDN 缓存规则

```ts
function netlifyContentCache() {
  return {
    headers: {
      'Cache-Control': 'public, max-age=0, must-revalidate',
      'Netlify-CDN-Cache-Control': 'public, max-age=600, must-revalidate',
      'Netlify-Vary': 'query,cookie=chen-blog-theme',
    },
  }
}

export default defineNuxtConfig({
  sitemap: {
    sources: ['/api/__sitemap__/urls'],
    cacheMaxAgeSeconds: 0,
  },
  routeRules: {
    '/': netlifyContentCache(),
    '/posts': netlifyContentCache(),
    '/posts/**': netlifyContentCache(),
    '/archive': netlifyContentCache(),
    '/about': netlifyContentCache(),
    '/categories/**': netlifyContentCache(),
    '/tags/**': netlifyContentCache(),
    '/rss.xml': netlifyContentCache(),
    '/sitemap.xml': netlifyContentCache(),
    '/robots.txt': netlifyContentCache(),
    '/api/public/**': netlifyContentCache(),
    '/api/**': { cache: false },
  },
})
```

内容页、公开读取 API、RSS、Sitemap 和 robots 只使用 Netlify Edge CDN 缓存 600 秒，浏览器始终重验证。到期后必须同步回源，不设置 `stale-while-revalidate` 或 `durable`，也不使用 `defineCachedEventHandler` 增加 Nitro 存储缓存；CMS 页面和写入 API 不缓存。发布后最多 10 分钟看到旧内容是 V1.0.0 的明确产品行为，不是故障。

公开页面的 SSR HTML 会读取 `chen-blog-theme` Cookie，因此缓存键必须同时保留全部 query 参数，并只针对这个主题 Cookie 分流：`Netlify-Vary: query,cookie=chen-blog-theme`。不得使用笼统的 `Vary: Cookie`，否则 Supabase 或其他无关 Cookie 会造成高基数缓存碎片；也不得只写 Cookie 维度而丢失 Nuxt/Netlify 已有的 query 分键。无 Cookie、`light` 和 `dark` 请求必须各自得到与其主题一致的 SSR HTML。公开 JSON API 复用相同的严格缓存函数，保持生产验证和最长陈旧时间只有一套规则。

V1.0.13 不对这些路由使用 Netlify preset 的数字 `isr`：当前 Nitro preset 会额外生成长时间 stale 窗口，不能实现“最多旧 10 分钟”的硬上限。`@nuxtjs/sitemap` 的内部 SWR 也通过 `cacheMaxAgeSeconds: 0` 关闭，避免与 Netlify CDN 叠加。

### 部署约定

- `blog-web` 部署为 Netlify 上支持 SSR/Nitro 的 Nuxt 应用。
- `cms-admin` 部署为独立 Netlify 静态站点。
- `www.example.com` 指向博客，`admin.example.com` 指向 CMS。
- 若使用 Cloudflare，只托管 DNS；不要开启代理缓存，避免 Netlify、Nitro、Cloudflare 三层同时缓存 HTML。
- `cms-admin` 的前端环境变量仅包含 Supabase URL 与现代 publishable key。`blog-web` 以 `NUXT_SUPABASE_PUBLISHABLE_KEY` 保存 Nitro 公开查询使用的低权限 key，以 `NUXT_SUPABASE_SERVICE_ROLE_KEY` 保存发布接口使用的现代 secret key；两者都不进入 `public` runtime config，任何 secret 都不进入浏览器构建产物。
- Supabase Auth Site URL 指向生产 CMS；发布稳定后禁用 legacy JWT API keys，不在代码、文档或日志中保存其值。
- 部署后以真实 `GET` 检查 Netlify `Cache-Status` 与 `Age`：命中时 `Age` 必须为 `0..600`，且响应不得包含 `stale-while-revalidate` 或 durable cache；`Netlify-Vary` 必须同时包含 `query` 与 `cookie=chen-blog-theme`。

## 12. 开发阶段与验收标准

### 阶段 1：项目与数据库基础

1. 初始化 pnpm workspace、Nuxt、Vue CMS 与 Supabase CLI。
2. 创建核心表、索引、约束和时间戳触发器。
3. 创建 RLS、Storage policy 与 `post-images` bucket。
4. 关闭公开注册，并创建唯一 owner。
5. 用 publishable key 分别以匿名与 owner 身份执行 Data API 冒烟测试，确认显式 `GRANT` 已使所需表可访问且 RLS 结果正确。

**验收：** 匿名用户和非 owner 登录用户都只能读取公开内容，无法读取草稿或已删除文章；owner 可以完成当前版本允许的内容管理操作；任何身份都不能直接 `DELETE posts`。

### 阶段 2：CMS 内容工作流

1. 登录、登出与修改密码；完整响应式壳层提供概览、页面上下文、公开博客入口、快捷新建、主题切换与 owner 菜单。
2. 文章工作区提供服务端搜索、状态/分类筛选、允许列表排序、exact-count 分页、结果计数、URL 条件恢复，以及桌面表格和移动内容行。
3. 文章草稿编辑、分类标签管理；CodeMirror 只编辑 Markdown/Comark 源文本，编辑页明确显示脏状态、保存/发布进度、正文统计、封面和发布检查，当前阶段不实现草稿最终预览。
4. 图片上传、媒体搜索与格式筛选、详情和公开 URL 复制、媒体库选择、插入 Markdown 图片、封面选择与文章软删除；不提供物理删除入口。
5. 所有数据工作区统一覆盖 skeleton、保留数据刷新、错误重试、真正空数据、筛选无结果和行级操作状态。
6. 保存与发布前统一执行 content-rules 校验；Nitro 发布接口与发布按钮。

**验收：** 管理员可在 320px、768px 与 1440px 下完成“定位草稿 → 上传或选择图片 → 编辑 Markdown → 保存并发布”的完整流程；列表筛选可恢复、分页总数正确，加载/空/错误状态互不混淆。发布时间不可由 CMS 指定为未来时间；Nitro 必须重新读取并校验最终保存的标题、Slug、正文与 Markdown 安全规则，不能发布空字段或非法 Slug；浏览器直连 Data API 无法将状态改为 `published`，也无法直接改动已发布文章的分类、标签关系或封面元数据；允许 Origin 的 `OPTIONS` 返回 `204`，未知 Origin 返回 `403`，无 token 与非 owner token 均不能发布。

### 阶段 3：公开博客

1. 首页、列表、详情、分类、标签、归档。
2. 统一 Comark 渲染、稳定标题锚点与文章目录、`@nuxt/image` 响应式图片、SEO、仅含摘要的 RSS 与 Sitemap。
3. 公开页面通过同源 Nitro DTO 接口读取内容，配置 Netlify 单层 CDN 严格 600 秒缓存并部署预览环境。

**验收：** 未发布或软删除文章无论通过页面、Nitro 接口还是直接调用 Data API 均不可访问；博客 HTML 和浏览器 JavaScript 不包含 `/rest/v1/` 查询或 Nitro 私有 publishable-key 配置，站内客户端导航只请求 `/api/public/**`；发布内容在最长 10 分钟内出现在前台；正文目录与实际 h2–h4 锚点一致，中文、内联格式和重复标题都能生成稳定且唯一的链接；公开文章的 `cover_media_id` 可解析为带宽高和 `sizes` 的响应式图片，Markdown 外部图片会被保存与发布校验拒绝；未被公开文章引用的媒体元数据不可被匿名读取。Netlify 预览中同时验证 Image CDN URL、窄屏尺寸和 GIF 行为，并验证 `light` / `dark` 主题 Cookie 命中相互隔离的缓存对象与正确 SSR 主题。

### 阶段 4：上线检查

1. 检查浏览器构建产物中没有 secret key。
2. 用匿名、普通已登录、owner 三种身份执行权限测试。
3. 检查 Storage 只能由 owner 上传；确认 V1.0.0 没有物理删除图片的接口或 policy。
4. 检查响应缓存头，确认没有超过 10 分钟的额外 HTML 缓存。
5. 配置生产域名、Supabase Auth Site URL 与允许的 redirect URL。
6. 执行 `supabase db advisors`，分级记录并处理 RLS、函数、暴露 schema 和 Auth 的 Advisor 结果，不得用“所有 warning 均已修复”代替实际结果。
7. 运行生产验证脚本，用同一唯一 URL 的连续请求证明 Netlify Edge `hit` 与 `Age <= 600`，并扫描博客和 CMS 浏览器 HTML/JavaScript，确认没有 `sb_secret_` 或服务端密钥变量。

Advisor 上线基线必须保留以下两项已评估结果：

- Security Advisor 当前唯一项是 `auth_leaked_password_protection` WARN。[Leaked Password Protection](https://supabase.com/docs/guides/auth/password-security#password-strength-and-leaked-password-protection) 仅在 Pro 及以上套餐可用；当前套餐下将它记录为 V1 接受的付费能力例外，不声称 Security Advisor 已无 warning。
- Performance Advisor 当前有 `posts_cover_media_id_idx` 与 `posts_category_id_idx` 两项 [`unused_index` INFO](https://supabase.com/docs/guides/database/database-linter?lint=0005_unused_index)。新项目数据量小、访问样本少时未使用索引属预期现象；前者支持封面媒体关联查询，后者支持全状态分类外键检查。两者都不因当前 INFO 直接删除，待有真实流量样本后再复评。

## 13. 后续演进顺序

V1.0.0 稳定后，按以下顺序增加能力：

1. 草稿预览（管理员鉴权 + `noindex`）。
2. 项目、友链、站点配置的后台管理。
3. 中文搜索方案验证与搜索索引。
4. 操作日志、导出和备份。
5. 发布 webhook 与精确缓存失效。
6. 定时发布与 Cron。
7. AI、邮件订阅及其他第三方服务。

任何新表都必须先补齐 RLS、索引、约束和测试，再接入 CMS 页面。
